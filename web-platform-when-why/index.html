The Web as a platform

The Web is awesome for several reasons:

The Web is not…

  • An operating system
  • A hardware specification
First, the Web is not an operating system nor a hardware specification, the Web is unique in nature.

The Web is…

  • An unprecedented effort of consensus
  • A virtual machine for HTML5
  • A free model of content distribution
Second, the web is an unprecedent effort of consensus which leads us to third, the goals of this agreement are two: interoperability —as webpages are running in an standardized virtual machine with several implementations (Web engines); and a free —as in free speech— model of content distribution.

Lack of friction

Source: Every Step Costs you 20% of Users

The other benefit is actually the great differentiator. The Web spreads through the Internet which is not a propietary marketplace but a completely free information network. The URL is viral, any place in the Internet can point you to any Web application without ever facing an installation. You click the link, you get working content.

So, what's stopping the Web?

Well, there are obvious reasons for not to choose the Web. For instance, if the platform does not cover your API needs. But, even when this is not a problem and having such exclusive and attractive features as the aforementioned interoperability and lack of friction features, the Web does not seem to fit our needs. Why?

User experience

The answer is user experience. In the beginning, the Web was not designed for mobile usage. We surfed the Web from our desktops, with powerful multicore computers and a big screen and reliable & fast network connections.

Dependency on connectivity

In the mobile realm you can not rely on always being connected. If the connectivity is lost, the experience is broken. The complete application vanishes. Of course a user can understand for an online application not tofor show content if offlibut to breaking with no information? I don't think so.

Dependency on network speed

Source: How Loading Time Affects Your Bottom Line

Not only is the lack of connectivity a problem but unreliable connection speeds are an issue as well.

Inability to re-engage the user

Source: Every Step Costs you 20% of Users

When browsing from the desktop, we are in a Web mindset paying attention to the browser and actively consuming information. In the mobile experience, we are not longer paying special attention to any not immediate task. We are used to receive notifications upon important things. Until now, this was something impossible to do for the Web loosing the opportunity to re-engage the user.

Performance

Source: Milennials and mobility, Oracle

And last but not least, there is more about perceived performance. Interoperability comes with a cost on translation overhead and this reduces the effective speed of the technology. In the Web, we need to be extra careful to keep things flowing. Given the single thread nature of JavaScript, if a computation takes so long, frames per second will drop under 60 fps giving the sensation of slow and jagged animations.

Let's bring the benefits of apps to the Web!

So, the idea is not to use core Web technologies inside native applications but to translate part of the mobile UX to the Web. How? Getting rid of connectivity and browser dependencies.

Service Workers

To do that, I want to introduce a technology you probably have already heard: Service Workers.

Service Workers are…

  • JavaScript programs
  • Run in a separate thread
  • Independent from the browser being open
  • Driven by events
As you probably know, a worker is a JS program that runs in a separated thread. The news about service workers is that they does not require the browser to be open at all. Once registered, the Service Worker will wake up to attend to specific events.

Functional Events

Those able to wake up the Service Worker

These events are now three: fetch, sync & push.

Fetch

The first one, fetch, is triggered each time a client page is performing an HTTP request for a resource or navigation. That is, it's a programmable proxy hijacking request to response is a customized way.
In addition to the fetch event, new cache API allow for storing pairs of HTTP request and responses. So we can go to network, retrieve the content, cache it and re-use from cache in the following requests.

Push

Contrary to fetch or sync events, push events originate from the network and are sent to the applications. After a client subscribe to the Push service, the client sends the subscription information to its remote server. Now when the server receive information for the client, it can communicate this info to the client through the Push server. Like SSE events but not requiring the app to be open.
Combined with the notification API, the Web app can visibly notify the user about some event.

Sync

Finally, sync events are triggered in two ways but currently only one is implemented in the browsers. The sync event happens each time the device regains connectivity. A sync event can serve to perform background operations such as receiving incoming events or synchronizing with servers.
